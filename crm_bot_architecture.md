# Архитектура Telegram-бота для CRM Битрикс24 с ИИ-интерфейсом

## 1. Цель и общая концепция

Проект — Telegram-бот, который выступает интеллектуальным интерфейсом к CRM Битрикс24 для команды партнёров. Бот добавляется в групповые чаты, где партнёры обращаются к нему через упоминание (`@bot_name`) и задают произвольные вопросы по состоянию CRM на естественном языке. Бот через REST API Битрикс24 получает данные, обрабатывает их через Claude (с использованием Kie.ai как провайдера токенов) и формирует ответ непосредственно в чате.

В качестве следующего этапа предусмотрено личное взаимодействие с ботом: у каждого партнёра в личных сообщениях с ботом будет доступно кнопочное меню для типовых отчётов (по лидам, по сделкам, по активности и так далее).

Доступ к административным функциям, в первую очередь к выбору используемой модели Claude, имеет только один пользователь — владелец проекта.

## 2. Ключевые архитектурные решения

Бот работает исключительно в режиме чтения CRM. Никакие модификации (создание, изменение, удаление сделок, лидов, контактов) не предусмотрены ни в MVP, ни в дальнейшем развитии — это критическое ограничение, снижающее риск ошибочных действий ИИ.

Доступ к Claude осуществляется через сервис Kie.ai, который проксирует нативный Anthropic Messages API по адресу `https://api.kie.ai/claude/v1/messages`. Формат запросов и ответов полностью совпадает с Anthropic, поэтому официальный Anthropic SDK можно использовать с подменой `base_url` и API-ключа. Все три текущие модели семейства Claude 4 доступны: Haiku 4.5, Sonnet 4.6 и Opus 4.7.

Используется prompt caching через `cache_control` на системном промпте и описаниях инструментов. Это ключевой механизм снижения стоимости запросов, поскольку статичная часть промпта (около пяти тысяч токенов) переиспользуется при каждом обращении партнёров.

Реализуется маршрутизация запросов между моделями: простые лукапы направляются на Haiku, аналитические вопросы на Sonnet, сложные многоступенчатые задачи на Opus. Маршрутизация выполняется по правилам (ключевым словам и шаблонам). Текущая «модель по умолчанию» и режим маршрутизации (включена/отключена/принудительно одна модель) задаются через админ-панель.

Изоляция данных между партнёрами обязательна. Каждый партнёр в БД связан с конкретным набором ответственных в Битрикс24, и все запросы к CRM автоматически фильтруются по этому набору. Партнёр не может увидеть сделки или лиды, которые ему не принадлежат.

## 3. Решение проблемы доступа к Telegram из РФ

Сервер размещён в РФ (Timeweb), и прямой доступ к `api.telegram.org` оттуда нестабилен. Применяется проверенная на предыдущих проектах схема: прокси используется исключительно для трафика Telegram-бота, остальные исходящие соединения сервера (Битрикс24, Kie.ai, обновления пакетов и так далее) идут напрямую. Это упрощает конфигурацию, не создаёт лишней зависимости от прокси для критичных сервисов и позволяет включать или отключать проксирование одной переменной окружения.

Настройка управляется единственной переменной `TELEGRAM_PROXY_URL`. Поддерживаются три варианта значения. HTTP или HTTPS-прокси с авторизацией задаётся в формате `http://login:password@host:port` — это предпочтительный вариант, поскольку не требует дополнительных зависимостей сверх стандартного aiohttp. SOCKS5-прокси задаётся как `socks5://login:password@host:port` либо `socks5h://login:password@host:port`; вариант с `socks5h` отличается тем, что разрешение DNS происходит на стороне прокси, что иногда необходимо при блокировках на уровне DNS у российского провайдера. Если переменная пустая или отсутствует, бот подключается к Telegram API напрямую без прокси — это используется для разработки и для запуска вне РФ.

В коде бот при старте считывает `TELEGRAM_PROXY_URL` и, если значение задано, создаёт сессию aiogram через `AiohttpSession` с указанным прокси:

```python
from aiogram import Bot
from aiogram.client.session.aiohttp import AiohttpSession

session = AiohttpSession(proxy=settings.TELEGRAM_PROXY_URL) if settings.TELEGRAM_PROXY_URL else AiohttpSession()
bot = Bot(token=settings.TELEGRAM_BOT_TOKEN, session=session)
```

Для поддержки SOCKS5 в зависимостях проекта обязательно присутствует пакет `aiohttp-socks`. Без него попытка использовать SOCKS-прокси приводит к ошибке `Missing dependencies for SOCKS support`.

Проверка доступности проводится двумя командами на сервере перед первым запуском. Сначала прямой доступ: `curl -I --max-time 10 https://api.telegram.org`. Если таймаут — проверка через прокси: `curl -I --max-time 15 -x "http://LOGIN:PASSWORD@IP:PORT" https://api.telegram.org` для HTTP или `curl --max-time 15 -x "socks5h://LOGIN:PASSWORD@IP:PORT" https://api.telegram.org` для SOCKS5. Корректный ответ — `HTTP/2 200` или аналогичный.

Доступ к Битрикс24 идёт напрямую: портал хостится в РФ, и сервер Timeweb обращается к нему без прокси. Доступ к `api.kie.ai` также по умолчанию настраивается напрямую — Timeweb обычно даёт стабильную связность с зарубежными ресурсами для API-трафика. Если в эксплуатации выявится нестабильность, для Kie вводится отдельная переменная `KIE_PROXY_URL` по той же схеме, что и `TELEGRAM_PROXY_URL`, без переиспользования telegram-прокси, чтобы не создавать единую точку отказа.

## 4. Технологический стек

Язык: Python 3.11 или новее. Асинхронная архитектура на базе `asyncio`.

Основные библиотеки: `aiogram 3.x` для Telegram, `aiohttp-socks` для поддержки SOCKS5-прокси при работе бота, `anthropic` (официальный SDK Anthropic, используется с подменой base_url под Kie), `aiohttp` для запросов к Битрикс24, `aiosqlite` для хранилища, `pydantic-settings` для конфигурации, `structlog` для журналирования, `python-dotenv` для секретов.

Хранилище данных: SQLite в файле на диске. Этого достаточно для нагрузки до нескольких десятков партнёров и нескольких тысяч запросов в день. Миграция на PostgreSQL предусмотрена на уровне абстракции репозиториев, но в MVP не реализуется.

## 6. Схема базы данных

```sql
-- Пользователи бота с маппингом на Битрикс24
CREATE TABLE users (
    telegram_id        INTEGER PRIMARY KEY,
    role               TEXT NOT NULL CHECK (role IN ('admin', 'partner', 'viewer')),
    b24_user_ids       TEXT,                  -- JSON-массив ID ответственных в Б24
    display_name       TEXT,
    is_active          INTEGER DEFAULT 1,
    created_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Глобальные настройки бота (key-value)
-- Ключи: current_model, routing_mode (auto|forced), default_model, thinking_enabled
CREATE TABLE settings (
    key                TEXT PRIMARY KEY,
    value              TEXT NOT NULL,
    updated_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Состояние диалогов (для поддержания контекста в группах)
CREATE TABLE sessions (
    telegram_id        INTEGER NOT NULL,
    chat_id            INTEGER NOT NULL,
    messages_json      TEXT NOT NULL,         -- последние N сообщений в формате Anthropic messages
    updated_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (telegram_id, chat_id)
);

-- Журнал всех запросов для аудита и аналитики
CREATE TABLE audit_log (
    id                 INTEGER PRIMARY KEY AUTOINCREMENT,
    telegram_id        INTEGER,
    chat_id            INTEGER,
    chat_type          TEXT,                  -- private | group | supergroup
    question           TEXT,
    model_used         TEXT,
    tools_called       TEXT,                  -- JSON-массив имён инструментов
    answer             TEXT,
    input_tokens       INTEGER,
    cached_input_tokens INTEGER,
    output_tokens      INTEGER,
    credits_consumed   REAL,                  -- из ответа Kie
    duration_ms        INTEGER,
    error              TEXT,
    created_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Опциональный кэш данных Битрикс24 (для часто запрашиваемых статичных данных)
CREATE TABLE b24_cache (
    cache_key          TEXT PRIMARY KEY,
    payload            TEXT NOT NULL,
    expires_at         TIMESTAMP NOT NULL
);

CREATE INDEX idx_audit_telegram_id_created ON audit_log(telegram_id, created_at);
CREATE INDEX idx_audit_created ON audit_log(created_at);
```

## 8. Интеграция с Битрикс24

Клиент работает поверх входящего вебхука. Базовый URL формируется как `B24_WEBHOOK_URL + method_name + .json`, параметры передаются в теле POST-запроса.

Ограничения, которые клиент обязан соблюдать. Лимит примерно два запроса в секунду на портал — реализуется через семафор или token bucket. Лимит пятидесяти записей на один вызов методов `*.list` — реализуется через автоматическую пагинацию с параметром `start`. Метод `batch` позволяет объединять до пятидесяти вызовов в одном HTTP-запросе и должен использоваться для массовых выгрузок.

Минимальный набор используемых методов: `crm.deal.list`, `crm.deal.get`, `crm.lead.list`, `crm.lead.get`, `crm.contact.list`, `crm.company.list`, `crm.activity.list`, `crm.dealcategory.stage.list` (для интерпретации стадий воронки), `user.get` (для отображения имён ответственных).

Все запросы проходят через фильтр прав пользователя. Если у партнёра в `b24_user_ids` указано, например, `[12, 34, 56]`, то к любому фильтру для `crm.*.list` автоматически добавляется условие `filter[ASSIGNED_BY_ID]=[12, 34, 56]`. Это критично для изоляции данных.

## 9. Интеграция с Kie.ai

Клиент инициализируется через официальный Anthropic SDK с указанием альтернативного `base_url`:

```python
from anthropic import AsyncAnthropic

client = AsyncAnthropic(
    api_key=settings.KIE_API_KEY,
    base_url=settings.KIE_BASE_URL,
)
```

Идентификаторы моделей: `claude-haiku-4-5`, `claude-sonnet-4-6`, `claude-opus-4-7`.

Системный промпт и описания инструментов передаются с `cache_control: {"type": "ephemeral"}` для активации prompt caching. Рекомендуется использовать кэширование на один час (`ttl: 1h`), поскольку рабочий день партнёров формирует длительные сессии активности. Стоимость записи в кэш с часовым TTL составляет двукратную цену входных токенов, но компенсируется уже после двух чтений.

Function calling используется в нативном формате Anthropic: список инструментов передаётся в параметре `tools`, ответ модели содержит блоки `tool_use`, обработчик выполняет инструмент и возвращает результат в формате `tool_result` для следующей итерации.

Цикл orchestrator выглядит так. Первый вызов с системным промптом, инструментами и сообщениями пользователя. Если `stop_reason == "tool_use"`, парсятся все `tool_use` блоки в ответе, последовательно или параллельно выполняются соответствующие обработчики, результаты добавляются в `messages` как `tool_result`, делается следующий вызов. Цикл продолжается до получения `stop_reason == "end_turn"` или достижения лимита итераций (рекомендуется не более десяти, защита от бесконечного цикла).

В ответе Kie присутствует поле `credits_consumed` — оно записывается в `audit_log` для отслеживания расхода. В блоке `usage` также присутствуют поля `cache_read_input_tokens` и `cache_creation_input_tokens` — они тоже сохраняются для метрик эффективности кэширования.

## 10. Спецификация инструментов для function calling

Минимальный набор инструментов для MVP, каждый с JSON Schema:

`get_deals(filter_by_stage, filter_by_date_from, filter_by_date_to, assigned_by_id, limit)` — возвращает список сделок с применением пользовательского фильтра.

`get_deal_details(deal_id)` — детальная информация по конкретной сделке: контакт, компания, сумма, стадия, история изменений, последние активности.

`get_leads(filter_by_status, filter_by_date_from, filter_by_date_to, limit)` — список лидов.

`get_pipeline_summary(date_from, date_to)` — агрегированная сводка по воронке: количество сделок и сумма по каждой стадии за период.

`get_user_activity_summary(b24_user_id, date_from, date_to)` — сводка активности конкретного менеджера за период: новые сделки, закрытые, средний цикл, конверсия.

`search_contacts_or_companies(query)` — поиск контакта или компании по имени, телефону, email.

`get_recent_activities(deal_id, contact_id, limit)` — последние активности по сделке или контакту.

Каждый инструмент в обработчике автоматически применяет фильтр по `b24_user_ids` из контекста текущего пользователя. Этот фильтр не передаётся моделью, а добавляется на уровне обработчика — модель в принципе не знает о существовании других партнёров и не может его обойти.

## 11. Маршрутизация моделей

Реализация через правила. В файле `ai/router.py` определяются три категории по ключевым словам и шаблонам:

Категория «простой лукап» (направляется на Haiku 4.5): запросы с ключевыми словами «сколько», «какой статус», «найди», «покажи задачи», «когда последний контакт», и подобные. Это вопросы с однозначным фактическим ответом без интерпретации.

Категория «аналитика» (направляется на Sonnet 4.6, это значение по умолчанию): запросы с ключевыми словами «проанализируй», «сравни», «динамика», «почему», «как изменилось», «интерпретируй», и подобные. Большинство реальных вопросов попадает сюда.

Категория «сложная аналитика» (направляется на Opus 4.7): запросы с ключевыми словами «найди системные причины», «дай стратегические рекомендации», «комплексный анализ», «прогноз», «многофакторный».

Админ через панель может переключить режим маршрутизации в одно из трёх состояний: автоматический по правилам, принудительно одна модель (Haiku / Sonnet / Opus), или авто с возможностью эскалации (если ответ Haiku недостаточен, повтор на Sonnet — резерв, в MVP не обязательно).

## 12. Системный промпт

Структура системного промпта (передаётся с включённым cache_control):

В первой части — роль и назначение бота, описание того, что бот является интерфейсом к CRM Битрикс24 и работает строго в режиме чтения.

Во второй части — динамические данные о текущем пользователе: его имя, роль, список ответственных, на чьи данные он имеет доступ. Эта часть не кэшируется и формируется отдельно для каждого партнёра — поэтому она помещается ПЕРЕД основной частью с cache_control или после неё в отдельном сообщении, чтобы не разрушать кэш.

В третьей части — справочник по схеме данных CRM компании: список стадий воронки с расшифровкой, типы активностей, обозначения источников лидов, ключевые кастомные поля, если есть.

В четвёртой части — правила поведения: использовать только данные из инструментов, никогда не выдумывать цифры, если данных недостаточно — честно сообщать об этом, не раскрывать данные за пределами доступа пользователя, отвечать кратко и по существу, числовые ответы оформлять корректно (с разделителями тысяч, единицами измерения, валютой).

В пятой части — описания всех инструментов с примерами использования.

Третья, четвёртая и пятая части являются статическими и попадают под кэширование. Вторая часть формируется заново для каждого пользователя.

## 13. Логика обработки сообщения

Шаг первый: получение события от Telegram. В групповом чате — фильтр по `entities.type == "mention"` и совпадению с именем бота. В личном чате — любое сообщение или команда от авторизованного пользователя.

Шаг второй: middleware авторизации обращается к БД и подтягивает запись пользователя. Если запись отсутствует или `is_active == 0`, отправляется отказ. Контекст пользователя (роль, `b24_user_ids`) добавляется в context для дальнейшей обработки.

Шаг третий: извлекается текст вопроса. Из mention в группе удаляется упоминание бота. Загружается история диалога из таблицы `sessions` (последние сообщения в рамках TTL).

Шаг четвёртый: router определяет модель на основе текста вопроса и текущих настроек.

Шаг пятый: формируется запрос к Kie. Системный промпт собирается из статической части (с cache_control) и динамической части пользователя. Передаётся история сообщений и описания инструментов.

Шаг шестой: orchestrator запускает цикл вызовов до получения финального ответа. Каждый вызов инструмента логируется.

Шаг седьмой: ответ модели сохраняется в `sessions` (для поддержания контекста) и в `audit_log` (для аудита и метрик).

Шаг восьмой: текст ответа форматируется (markdown поддерживается Telegram-ботом) и отправляется в исходный чат как reply на сообщение пользователя.

## 14. Админ-панель

Доступ строго по `telegram_id == ADMIN_TELEGRAM_ID` из переменных окружения, без возможности расширения списка администраторов через интерфейс (защита от компрометации).

Главное меню админа открывается командой `/admin` в личном чате с ботом. Реализуется через inline-клавиатуру с следующими кнопками:

«Текущая модель» — показывает выбранную модель и режим маршрутизации, ниже три кнопки выбора модели (Haiku / Sonnet / Opus) и кнопка переключения режима маршрутизации (Auto / Forced).

«Статистика» — показывает за выбранный период (сегодня, неделя, месяц): общее число запросов, распределение по моделям, среднее число токенов, cache hit rate, общий расход credits.

«Пользователи» — список зарегистрированных партнёров с их telegram_id, отображаемым именем и привязанными `b24_user_ids`. Кнопки добавления, редактирования, деактивации.

«Последние ошибки» — последние десять записей `audit_log` с ненулевым полем `error`.

«Баланс Kie» — текущий остаток credits через вызов `GET /api/v1/chat/credit` к Kie.

«Тестовый запрос» — позволяет админу отправить тестовый вопрос и увидеть детальный лог: какая модель выбрана, какие инструменты вызваны, какой расход токенов.

Все изменения настроек через админ-панель сохраняются в таблицу `settings` и применяются мгновенно без перезапуска бота.

## 15. Партнёрский интерфейс в личных сообщениях (этап 2)

В личных сообщениях с ботом партнёр видит главное меню с inline-кнопками:

«Сделки в работе» — выводит список открытых сделок партнёра с краткой сводкой.

«Отчёт по лидам» — позволяет выбрать период (сегодня / неделя / месяц) и получить отчёт по новым лидам, их источникам, конверсии.

«Активность за период» — задачи, звонки, встречи за период.

«Воронка» — текущее состояние воронки по сделкам партнёра.

«Свободный вопрос» — переводит в режим текстового ввода, далее обработка как в группах.

Каждая кнопка триггерит соответствующий шаблонный запрос к боту, который проходит через тот же orchestrator. Это означает, что для типовых отчётов также можно применять кэширование промптов и роутинг моделей (большинство таких запросов прекрасно обрабатывается Haiku).

## 16. Безопасность и приватность

Все секреты (токен Telegram, ключ Kie, вебхук Б24) хранятся только в `.env`, не попадают в репозиторий. Файл `.env` исключён через `.gitignore`. На сервере права доступа `chmod 600`.

API-ключ Kie настраивается с IP-белым списком, в который добавляется статический IP сервера Timeweb (исходящий трафик к Kie идёт напрямую, не через telegram-прокси). Если в дальнейшем для Kie также включается прокси через `KIE_PROXY_URL`, в белый список добавляется IP прокси-сервера вместо IP Timeweb.

Логи запросов в `audit_log` содержат тексты вопросов и ответов, в которых может быть коммерческая информация. Доступ к файлу БД ограничивается на уровне ОС. Автоматическая ротация и очистка `audit_log` старше девяноста дней через периодическую задачу.

Rate limiting на уровне пользователя: не более десяти запросов в минуту от одного `telegram_id`. Защита от спама и от непреднамеренной утечки токенов.

Никаких write-операций в CRM не предусмотрено. Это архитектурное ограничение: даже если модель попытается запросить такое действие через инструмент, такой инструмент в принципе не зарегистрирован.

Передача коммерческих данных через инфраструктуру Kie (двухмесячное хранение логов на их стороне) принимается как осознанный компромисс; для повышенного класса конфиденциальности можно перевести часть критичных запросов на прямой Anthropic API через тот же SDK — для этого в `ai/client.py` достаточно поддержать второй клиент с альтернативной конфигурацией.

## 17. План реализации по этапам

Этап 1 — MVP, ориентировочно одна-две недели. Базовая структура проекта, конфигурация, БД с миграциями. Telegram-бот через прокси, реагирует на mentions в одном тестовом групповом чате. Клиент Битрикс24 с двумя-тремя ключевыми методами. Один вызов Claude через Kie без function calling, простая передача результатов CRM в промпт. Админ-команда `/set_model` и базовая `/stats`.

Этап 2 — полная функциональность, ориентировочно две-три недели. Полный набор инструментов для function calling. Orchestrator с циклом `tool_use`. Маршрутизация моделей через router. Авторизация по ролям с фильтрацией по `b24_user_ids`. Полноценная админ-панель с inline-кнопками.

Этап 3 — оптимизация и продакшен, ориентировочно одна неделя. Prompt caching с проверкой эффективности по метрикам. Аудит-логи и дашборд метрик. Корректная обработка ошибок и retries с экспоненциальной задержкой. Документация по эксплуатации.

Этап 4 — партнёрский интерфейс в личных сообщениях, ориентировочно одна-две недели. Кнопочное меню с шаблонными отчётами. Регистрация партнёров через админ-панель.

## 18. Развёртывание на сервере Timeweb

Подготовка сервера: Ubuntu 22.04 LTS, установка Python 3.11, sqlite3, git. Создание системного пользователя `crmbot` без прав sudo.

Развёртывание кода в `/opt/crm-bot/`. Создание виртуального окружения, установка `requirements.txt`. Заполнение `.env`. Запуск миграций БД (`python scripts/init_db.py`).

Создание systemd unit `/etc/systemd/system/crm-bot.service` с автоматическим перезапуском при падении и зависимостью от сети. Логи через journald.

Настройка ротации логов и периодических задач (cron или systemd timer) для очистки старых записей `audit_log`.

Регулярные бэкапы файла SQLite на отдельное хранилище — еженощно через `sqlite3 bot.sqlite ".backup /backup/path"`.

Мониторинг доступности через простой healthcheck-эндпоинт (если бот развёрнут с webhook) или периодическую проверку `journalctl`.

## 19. Что нужно проверить на пилоте перед продуктовым запуском

Реальный тариф Kie на кэшированные чтения — сравнить `credits_consumed` на серии идентичных запросов с холодным и прогретым кэшем, удостовериться, что скидка применяется. Этот пункт определяет реальную экономику проекта.

Корректность маршрутизации Kie к запрошенным моделям — в публичных примерах документации Kie в ответе на запрос `claude-opus-4-7` поле `model` показывает значение `claude-opus-4-5-20251101`, что выглядит как устаревший пример, но требует фактической проверки на живом запросе. Если Kie скрытно подменяет модель — это меняет картину качества и стоимости.

Стабильность прямого подключения Timeweb к `api.kie.ai` под продолжительной нагрузкой — провести серию из нескольких сотен запросов в течение рабочего дня и зафиксировать процент отказов и таймаутов. Если стабильность ниже 99%, ввести `KIE_PROXY_URL` и направить Kie-трафик через прокси по аналогии с Telegram.

Стабильность telegram-прокси под пиковыми утренними сессиями, когда несколько партнёров одновременно обращаются к боту в групповых чатах.

Корректность фильтрации данных по `b24_user_ids` — обязательно ручное тестирование от лица партнёра с ограниченным доступом, проверка, что чужие сделки не возвращаются ни через прямой запрос, ни через косвенный (например, через имя контакта).

Поведение при отказах внешних сервисов — что происходит, если Kie вернул 429 или 500, если Битрикс24 недоступен, если прокси упал. Бот должен корректно сообщать о проблеме пользователю, а не молчать.
